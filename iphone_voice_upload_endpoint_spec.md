# iPhone Shortcut Audio Upload Endpoint Specification

## Purpose

Add a **second route** to the existing API that accepts an audio recording uploaded by an iPhone Shortcut, transcribes the speech, parses the user's intent, optionally executes downstream actions, and returns a compact JSON response that the Shortcut can display.

This route should be designed so it can evolve independently from the existing text-based route, while still reusing shared internal services such as authentication, transcription, intent parsing, action dispatching, logging, and observability.

---

## High-Level Goals

1. Accept a short audio file recorded on iPhone via Apple Shortcuts.
2. Authenticate the mobile client securely.
3. Persist request metadata and optionally the raw audio for debugging/auditing.
4. Transcribe the uploaded audio to text.
5. Parse the transcribed text into structured intent.
6. Execute the intended action when confidence and validation rules allow it.
7. Return a concise JSON payload that works well in Apple Shortcuts.
8. Keep the route modular so it can later support:
   - different audio formats
   - different speech-to-text providers
   - async/background processing if ever needed
   - multiple client types beyond iPhone Shortcuts

---

## Recommended Route

### Primary route

`POST /v1/voice/commands`

### Why this path

- Keeps the new feature separate from any existing text-command route.
- Leaves room for future related routes such as:
  - `POST /v1/voice/transcribe`
  - `POST /v1/voice/commands/preview`
  - `GET /v1/voice/commands/{id}`

### Alternative acceptable paths

- `POST /v1/shortcut/voice-command`
- `POST /v1/ingest/audio-command`
- `POST /v1/voice-command`

Recommended choice remains:

`POST /v1/voice/commands`

---

## Request Contract

### Content type

Use:

`multipart/form-data`

This is the most practical format for an iPhone Shortcut uploading an audio file together with metadata.

### Request parts

#### Required fields

- `audio` — uploaded audio file

#### Optional metadata fields

- `client` — string, e.g. `iphone_shortcut`
- `device_id` — stable device identifier generated by client
- `shortcut_version` — version string of the installed shortcut
- `timezone` — IANA timezone, e.g. `Europe/Vienna`
- `locale` — e.g. `de-AT` or `en-US`
- `submitted_at` — ISO-8601 timestamp from device
- `request_id` — client-generated UUID for idempotency/tracing
- `mode` — e.g. `execute` or `dry_run`
- `return_debug` — boolean-like string flag for development only

### Authentication

Use bearer token authentication.

#### Header

`Authorization: Bearer <mobile_api_token>`

### Additional recommended headers

- `X-Client-Platform: ios-shortcuts`
- `X-Client-Version: 1.0.0`
- `X-Request-Id: <uuid>`

These are optional, but useful for logging and debugging.

## Example Request

```bash
curl -X POST "https://api.example.com/v1/voice/commands" \
  -H "Authorization: Bearer YOUR_MOBILE_TOKEN" \
  -F "audio=@voice-command.m4a" \
  -F "client=iphone_shortcut" \
  -F "timezone=Europe/Vienna" \
  -F "locale=de-AT" \
  -F "submitted_at=2026-03-15T09:12:44+01:00" \
  -F "request_id=4c8c4c36-b02a-4d13-8f3f-ec4e0b2c2f1f" \
  -F "mode=execute"
```

## Expected Audio Characteristics

The endpoint should be tolerant and not overly strict in v1.

### Supported formats

Accept at least:

- `.m4a`
- `.mp3`
- `.wav`
- `.aac`
- `.mp4` audio-only if produced by client tooling

### File size limits

Recommended initial limit:

- default max file size: **10 MB**
- ideal expected size: **under 2 MB**

This is more than sufficient for a few seconds of speech.

### Duration guidance

Recommended expected duration:

- target: 2–15 seconds
- soft limit: 30 seconds
- hard reject above: configurable, e.g. 120 seconds

### Validation behavior

Reject when:

- no file provided
- unsupported media type
- empty file
- file size exceeds configured limit
- MIME type and extension both appear invalid

Do not reject purely because codec detection is imperfect if the transcription provider can still process the file.

## Response Design Principles

The Apple Shortcut should receive a response that is:

- fast to parse
- stable over time
- human-readable
- machine-readable
- safe to show directly to the user

The top-level response should always include:

- success state
- transcript or transcription status
- intent/action result
- user-facing summary text
- request identifier

## HTTP Status Codes

Use HTTP status codes meaningfully, but keep the JSON body consistent.

### Recommended mapping

- `200 OK` — request processed successfully, even if downstream action was intentionally skipped
- `400 Bad Request` — malformed form data or missing required fields
- `401 Unauthorized` — missing or invalid token
- `403 Forbidden` — authenticated but not allowed to use this route
- `413 Payload Too Large` — file exceeds limit
- `415 Unsupported Media Type` — unsupported file type
- `422 Unprocessable Entity` — transcription or validation failure after parsing
- `429 Too Many Requests` — rate limit exceeded
- `500 Internal Server Error` — unexpected server-side error
- `502/503/504` — upstream provider outage/timeouts where appropriate

### Important implementation note

Even for non-200 responses, return the same general JSON envelope shape where feasible so the Shortcut can show a useful human-facing error message.

## Execution Modes

Support two modes from day one.

### 1. `execute`

The server performs the downstream action.

Example:
- creates Todoist task
- sends Slack message
- creates calendar event

### 2. `dry_run`

The server transcribes and parses intent, but does not execute.

Use cases:
- debugging
- confidence validation
- safe testing during rollout

### Behavior

If `mode` is omitted, default to:

`execute`

## Endpoint Lifecycle

### Step 1: Authenticate request

- Validate bearer token
- Resolve user/account context
- Attach authorization scopes to request context

### Step 2: Parse multipart form

- extract audio file
- extract optional metadata
- generate server-side request id if not supplied

### Step 3: Validate input

- check file exists
- check size
- inspect content type / extension
- optionally estimate duration

### Step 4: Persist raw request metadata

Store at minimum:

- request id
- user id
- timestamp
- filename
- content type
- file size
- client metadata
- status = `received`

### Step 5: Store audio temporarily or permanently

Recommended v1:

- save to temp storage first
- optionally upload to object storage if logging/audit enabled

### Step 6: Transcribe audio

Call a transcription service abstraction.

Input:
- file path or bytes
- locale hint
- user language hint if available

Output:
- transcript text
- provider metadata
- confidence if available
- duration if available

### Step 7: Normalize transcript

Examples:
- trim whitespace
- collapse duplicate spaces
- normalize punctuation lightly
- preserve original transcript separately if needed

### Step 8: Parse intent

Pass transcript into your existing intent parser.

Expected output shape:

```json
{
  "service": "todoist",
  "action": "create_task",
  "entities": {
    "content": "send the proposal",
    "due": "2026-03-16T09:00:00+01:00"
  },
  "confidence": 0.96,
  "needs_confirmation": false
}
```

### Step 9: Policy and validation gate

Before executing, validate:

- user authorized for target integration
- required entities present
- confidence above threshold
- command not ambiguous beyond allowed tolerance

### Step 10: Execute downstream action

Dispatch to existing service adapters.

Examples:
- Todoist adapter
- Slack adapter
- Calendar adapter

### Step 11: Persist final outcome

Store:

- transcript
n- normalized transcript
- parsed intent
- action result
- provider references
- error payload if any
- latency breakdown
- final status

### Step 12: Return Shortcut-friendly response

Include concise `result_text` for direct display on device.

## Recommended Idempotency Behavior

Because mobile networks can be flaky and users may retry, support idempotency where possible.

### Approach

If `request_id` is provided:
- store it with unique constraint per user
- if duplicate request arrives within a safe time window, return previous result instead of re-executing

This is especially useful for non-idempotent actions like sending Slack messages.

---

## Confidence and Safety Rules

The endpoint should not blindly execute every transcript.

### Recommended rules

- If transcript is empty: do not execute
- If intent confidence below threshold: do not execute
- If required entities are missing: do not execute
- If action target is disallowed for user: do not execute
- If downstream integration is disconnected: do not execute

### Response in these cases

Return `ok: true` or `ok: false` depending on your preference, but include a clear `result_text`, such as:

- `I heard the request, but it was too ambiguous to execute safely.`
- `Todoist is not connected for this user.`

Recommended execution status:

`skipped`

## Security Requirements

### Authentication

- Require bearer auth
- Do not expose anonymous upload access

### Authorization

Check user-level permissions/scopes for:
- voice command use
- integration access
- specific action types if relevant

### File handling

- never trust client filename
- sanitize filenames
- validate content type conservatively
- store temp files outside public web root
- delete temp files after processing when not needed

### Abuse protection

- apply per-user rate limit
- apply per-token rate limit
- optionally apply IP-based rate limit

### Sensitive data handling

Audio and transcript may contain sensitive information. Treat as confidential.

Recommended controls:
- encryption at rest for stored audio/transcripts
- strict log redaction where needed
- configurable retention policy

---

## Retention Policy

Make this configurable.

### Recommended v1 defaults

- raw audio temp file: delete after successful processing unless debug retention enabled
- persisted audio object: optional, default off in production if not required
- transcript + structured result: retain for product/debugging needs
- error cases: optionally retain raw audio for a short window, e.g. 7 days, only if compliant with your privacy requirements

## FastAPI Implementation Notes

### Dependency suggestions

Create explicit dependencies such as:

- `require_bearer_auth`
- `get_voice_command_service`
- `get_transcription_service`
- `get_command_dispatcher`
- `get_clock`
- `get_request_id_generator`

### Avoid

- putting transcription logic directly in router
- putting provider SDK calls directly in route function
- mixing persistence and parsing logic in controller code

### Prefer

A single orchestration service, e.g.:

```python
result = await voice_command_service.handle_uploaded_audio(
    user=current_user,
    audio_upload=audio,
    metadata=VoiceCommandRequestContext(...),
)
```

## Downstream Integration Contract

The voice endpoint should not know implementation details of Todoist, Slack, Calendar, etc.

Instead it should dispatch to an internal abstraction such as:

```python
class CommandDispatcher:
    async def dispatch(self, parsed_intent: ParsedIntent, user: AuthenticatedUser) -> DispatchResult:
        ...
```

This keeps the endpoint reusable and maintainable.

## Apple Shortcut Compatibility Notes

The endpoint response should be optimized for Apple Shortcuts.

### Strong recommendation

Always include:

- `ok`
- `result_text`
- `request_id`

Even if everything else changes later, these three fields should remain stable.

### Why

The Shortcut can simply:
1. parse JSON
2. extract `result_text`
3. display it with `Show Result`

This keeps the client workflow very simple and durable.

## Recommended Non-Functional Requirements

### Performance targets for short audio clips

For clips of a few seconds on normal mobile connectivity, target server-side timing roughly like:

- upload parsing: under 150 ms
- transcription: 500–2500 ms
- intent parse: 300–1000 ms
- action execution: 200–2000 ms

### Overall API target

Aim for:
- p50 under 3 seconds
- p95 under 7 seconds

for short commands, excluding user recording time on device.

## Suggested Acceptance Criteria

1. The endpoint accepts an `.m4a` file from Apple Shortcuts using `multipart/form-data`.
2. The request is authenticated with bearer token.
3. The server transcribes the audio successfully for clear short speech.
4. The transcript is passed into the existing intent parsing flow.
5. In `execute` mode, a valid parsed command triggers the correct downstream adapter.
6. In `dry_run` mode, no downstream side effect occurs.
7. The response always includes `ok`, `request_id`, and `result_text`.
8. Errors return stable JSON and appropriate HTTP status codes.
9. Request lifecycle and latency are logged in structured form.
10. The implementation is modular and does not embed provider logic directly in the FastAPI route.
